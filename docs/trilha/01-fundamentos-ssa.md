# Module 1 - SSA Fundamentals

## Learning outcomes

At the end of this module, you should be able to:

- Explain the shift from syntax-centered development to meaning-centered architecture.
- Distinguish SSA responsibilities from AI-assisted development and infrastructure engineering.
- Convert an ambiguous problem statement into a semantic architecture draft.

## Core concepts

### 1. Historical transition of computing

- Classical model: `human -> code -> machine`.
- AI-mediated model: `human -> conceptual model -> AI -> code -> machine`.
- SSA works primarily in the conceptual layer.

### 2. Semantic interface

- Language is not documentation only; it is execution design.
- Intentions, policies, constraints, and goals are explicit engineering artifacts.

### 3. SSA role boundaries

- SSA: models behavior, contracts, and decision logic.
- AI-assisted dev: implements and integrates.
- Infra dev: provides runtime, security, and operability foundations.

## Lesson sequence

### Lesson 1 - From programming to semantic architecture

- Goal: build a clear mental model for the new discipline.
- Class activity: analyze one legacy system and one AI-native system.
- Output: a one-page comparison table with design implications.

### Lesson 2 - Ontology of SSA work

- Goal: formalize concepts such as semantic model, behavioral contract, and policy layer.
- Class activity: map a current product decision into semantic components.
- Output: a first SSA architecture canvas.

### Lesson 3 - Problem reframing

- Goal: transform vague requests into explicit intention structures.
- Class activity: rewrite three ambiguous prompts as architecture-level specs.
- Output: reframed briefs with acceptance criteria.

## Applications

1. **Application A - Current-state diagnosis**
- Choose a real team/system.
- Identify 10 implicit decisions currently hidden in code or tribal knowledge.
- Recast each decision as a semantic contract.

2. **Application B - SSA kickoff document**
- Produce `SSA Fundamentals in Context` with:
- System objective
- Stakeholders and constraints
- Initial architecture hypotheses
- Top 5 semantic risks

## Assessment rubric (0-2 each)

- Conceptual clarity.
- Quality of reframing.
- Separation of intention/policy/execution.
- Traceability of decisions.
- Practical applicability.

Passing threshold: **8/10**.
